# 미스터 노우 Toss 미니앱 설계

## 배경

미스터 노우는 Toss 미니앱으로 제공하는 판정형 갈등 분석 앱이다. 사용자는 대화 텍스트, 캡처 이미지, 녹음, 녹음 파일을 제출하고, 귀여운 심판 캐릭터에게 "누가 얼마나 잘못했는지"를 퍼센티지로 판정받는다.

참고 자료:

- Toss Apps in Toss 미니앱 소개: https://toss.im/apps-in-toss/blog/making-miniapps
- Toss Apps in Toss 개발자 문서: https://developers-apps-in-toss.toss.im/
- Hada News 목표 참고 글: https://news.hada.io/topic?id=29639
- 검수 참고 도구: https://github.com/garrytan/gstack
- 개발 템플릿 참고: https://github.com/sooin123456/Lua_template

## 목표

- Toss WebView 미니앱으로 실행 가능한 MVP를 만든다.
- 사용자가 네 가지 입력 방식 중 하나를 선택해 상황을 제출할 수 있게 한다.
- 입력 내용은 분석 전에 항상 텍스트 확인 화면을 거치게 한다.
- 결과는 오늘의 판정, A/B 잘못 퍼센티지, 근거 3개, 한 줄 조언으로 보여준다.
- 기본 판정은 규칙 기반으로 동작하게 하고, 추후 LLM 분석기로 교체할 수 있게 한다.
- 캡처본은 MVP부터 OCR 자동 추출을 목표로 한다.
- 판정형 재미를 유지하되, 과도한 단정이나 위험 상황 방치를 피한다.

## 비목표

- 첫 버전에서 로그인, 결제, DB 저장, 이메일, 소셜 기능은 만들지 않는다.
- 사용자 대화, 캡처, 음성 파일을 기본 저장하지 않는다.
- 법률 판단, 의학적 판단, 심리 진단, 실제 중재 결과 보장을 제공하지 않는다.
- `Lua_template`을 그대로 복사하지 않는다. 구조와 테스트 관점만 참고한다.

## 제품 경험

앱 이름은 **미스터 노우**다. 첫 화면은 귀여운 심판 캐릭터와 함께 네 개의 입력 카드를 보여준다.

입력 카드:

1. **카톡 싸움 붙여넣기**
   - 텍스트 복사 붙여넣기용이다.
   - 사용자가 채팅 내용이나 상황 설명을 바로 붙여넣는다.

2. **증거 캡처 제출하기**
   - 캡처본 붙여넣기 또는 이미지 업로드용이다.
   - OCR로 텍스트를 자동 추출한 뒤 사용자가 확인하고 수정한다.

3. **현장 녹음 시작**
   - 앱 안에서 녹음하는 흐름이다.
   - 녹음 후에는 텍스트 변환 결과를 확인하는 화면으로 이동한다.

4. **녹음 파일 불러오기**
   - 외부 녹음 파일 업로드용이다.
   - 파일 처리 후에는 텍스트 변환 결과를 확인하는 화면으로 이동한다.

모든 입력 방식은 `입력 선택 -> 텍스트 확인/수정 -> 분석 -> 결과` 흐름을 따른다. 자동 추출 또는 변환 결과가 있더라도 분석 전에 사용자가 최종 텍스트를 확인해야 한다.

## 결과 화면

결과 화면은 다음 정보를 고정 순서로 보여준다.

- 오늘의 판정
- A/B 잘못 퍼센티지
- 근거 3개
- 한 줄 조언

예시 구조:

- 오늘의 판정: "A가 조금 더 선 넘었어요"
- 퍼센티지: "A 68%, B 32%"
- 근거 3개:
  - 상대 감정보다 자기 억울함을 먼저 강조했다.
  - 사과보다 반박을 반복했다.
  - 문제 해결 제안이 늦게 나왔다.
- 한 줄 조언: "먼저 '내 말투가 세긴 했어'라고 시작해보세요."

퍼센티지는 항상 합이 100이 되도록 계산한다. 결과 문구는 "입력된 내용 기준의 재미용 판정"이라는 의미를 포함한다.

## 기술 아키텍처

프로젝트는 Toss 공식 WebView 미니앱 스캐폴드인 `create-ait-app` 기반으로 시작한다. 프론트엔드는 React와 TypeScript를 사용하고, 모바일 단일 화면 흐름을 우선한다.

권장 모듈:

- `app/routes` 또는 스캐폴드가 제공하는 라우팅 구조
- `features/input` 입력 방식 선택과 텍스트 확인 화면
- `features/result` 분석 결과 화면
- `features/analyzer` 판정 엔진 인터페이스와 구현체
- `features/ocr` OCR 어댑터
- `features/audio` 녹음 및 음성 파일 입력 어댑터
- `features/safety` 위험 신호 감지와 안전 문구
- `shared/ui` 공용 UI 컴포넌트

`Lua_template`은 인증, DB, 결제, 이메일 기능이 포함된 대형 웹앱 구조라 MVP 시작점으로는 쓰지 않는다. 다만 기능별 폴더 분리, e2e 테스트 구성, UI 컴포넌트 분리 방식은 참고한다.

## 분석 엔진

분석 엔진은 하이브리드 구조로 설계한다.

공통 인터페이스:

```ts
type JudgmentResult = {
  verdict: string;
  partyAPercent: number;
  partyBPercent: number;
  reasons: [string, string, string];
  advice: string;
  safetyLevel: "normal" | "caution" | "urgent";
};

type Analyzer = {
  analyze(input: { text: string; partyALabel?: string; partyBLabel?: string }): Promise<JudgmentResult>;
};
```

MVP 기본 구현은 규칙 기반 분석기다. 키워드, 사과 여부, 비난 표현, 회피 표현, 위협 표현, 감정 무시 표현, 해결 제안 여부를 점수화한다. LLM 분석기는 같은 인터페이스를 구현하는 선택 구현체로 둔다. API 키나 서버 설정이 없으면 규칙 기반 분석기가 기본값이다.

## OCR 설계

OCR은 MVP부터 자동 추출을 목표로 한다. 사용자는 이미지를 붙여넣거나 업로드할 수 있고, 앱은 OCR 어댑터를 통해 텍스트를 추출한다.

OCR 처리 후에는 항상 텍스트 확인 화면을 보여준다. 사용자는 오인식된 이름, 말풍선 순서, 이모지, 줄바꿈을 수정할 수 있다. OCR이 실패하면 이미지 미리보기와 함께 직접 입력할 수 있는 텍스트 영역을 보여준다.

MVP의 OCR 구현은 브라우저 내 처리로 시작한다. 서버로 이미지를 보내지 않는 방향을 기본값으로 두어 개인정보 부담을 줄이고, 필요하면 후속 버전에서 서버 OCR 어댑터를 같은 인터페이스 뒤에 추가한다. 구현 후보는 `tesseract.js`이며, 번들 크기와 Toss WebView 성능을 확인한 뒤 worker lazy-load로 붙인다.

## 음성 입력 설계

`현장 녹음 시작`은 마이크 권한을 요청하고 녹음 UI를 보여준다. `녹음 파일 불러오기`는 오디오 파일 선택을 제공한다. MVP에서는 녹음 경험과 파일 입력 흐름을 완성하고, 분석 전 텍스트 확인 화면에서 사용자가 대화 내용을 직접 입력하거나 붙여넣게 한다.

STT는 후속 기능으로 분리한다. `features/audio`에는 `TranscriptionAdapter` 인터페이스만 두고, MVP 기본 구현은 "수동 텍스트 확인" 어댑터다. 후속 버전에서 Web Speech API 또는 서버 STT를 붙여도 입력 이후 흐름은 바뀌지 않는다.

## 안전 정책

미스터 노우는 판정형 앱이지만 실제 법적, 의학적, 심리적 결론을 내리지 않는다. 결과는 입력된 내용 기준의 재미용 판정으로 표현한다.

위험 신호가 감지되면 일반 승패 판정보다 안전 안내를 우선한다. 위험 신호에는 폭력, 협박, 자해, 스토킹, 불법 촬영, 강압, 반복적 통제, 미성년자 위험 상황이 포함된다. 이 경우 결과 화면은 "누가 더 잘못했는지"보다 "지금은 안전 확보가 먼저"라는 안내와 도움 요청 문구를 보여준다.

개인정보 원칙:

- 기본 MVP에서는 입력 텍스트, 이미지, 음성 파일을 저장하지 않는다.
- 자동 추출 결과는 분석 전 사용자가 확인한다.
- 권한 요청은 필요한 순간에만 한다.
- 캡처와 음성에는 민감정보가 포함될 수 있음을 짧게 안내한다.

## 테스트 및 검수

기본 검증 흐름:

- 카톡 싸움 붙여넣기: 텍스트 입력부터 결과 표시까지 확인
- 증거 캡처 제출하기: 이미지 입력, OCR 결과 확인, 분석까지 확인
- 현장 녹음 시작: 권한 요청, 녹음 흐름, 텍스트 확인 화면까지 확인
- 녹음 파일 불러오기: 파일 선택, 텍스트 확인 화면, 분석까지 확인

자동 테스트:

- 규칙 기반 분석기의 퍼센티지 합이 항상 100인지 확인
- 근거가 정확히 3개인지 확인
- 한 줄 조언이 비어 있지 않은지 확인
- 위험 신호 입력에서 `safetyLevel`이 `caution` 또는 `urgent`로 올라가는지 확인
- OCR 실패 시 직접 입력 폴백이 보이는지 확인

검수 체크:

- `npm run lint`
- `npm test`
- 모바일 브라우저 수동 확인
- Toss WebView 실행 확인
- gstack 기반 구조 및 사용자 흐름 검수

## 구현 순서

1. `create-ait-app`으로 Toss WebView 미니앱 스캐폴드 생성
2. 첫 화면과 네 입력 카드 구현
3. 텍스트 확인/수정 화면 구현
4. 규칙 기반 분석기와 결과 스키마 구현
5. 결과 화면 구현
6. OCR 어댑터와 캡처 입력 흐름 구현
7. 음성 입력 어댑터와 녹음/파일 입력 흐름 구현
8. 안전 신호 감지와 안전 안내 구현
9. 단위 테스트와 주요 흐름 테스트 추가
10. gstack 검수 실행 및 피드백 반영
