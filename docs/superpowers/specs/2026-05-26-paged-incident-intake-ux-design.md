# 페이지 전환형 사건 접수 UX 설계

## 배경

현재 `누가 잘못 AI`는 홈, 입력, AI 사건 접수, 판독 결과, 보상 추천, 핫 Battle, 실시간 판정방까지 주요 기능이 붙어 있다. 하지만 홈 이후의 핵심 워크플로우는 아직 한 화면에 기능이 누적되는 느낌이 강하다. 특히 최근 추가된 `루아 사건 접수실`은 큰 입력창 아래에 사건 카드가 이어 붙는 구조라, 사용자가 “다음 단계로 진행하고 있다”는 감각이 약하다.

이번 개선의 목표는 전체 경험을 `홈 -> 자료 넣기 -> 루아 정리 중 -> 싸움 일지 확인 -> 판독 결과`로 재구성하는 것이다. 각 화면은 하나의 목적과 하나의 강한 CTA만 가진다. 사용자는 카톡, 캡처, 녹음 자료를 그대로 넣고, 루아가 자동으로 정리한 싸움 일지를 확인한 뒤 판독을 시작한다.

## UX 방향

선택한 방향은 `빠른 입력 + 자동 일지 + 페이지 전환형`이다.

사용자는 처음부터 사건을 잘 정리하지 않아도 된다. 홈에서 입력 방식을 고르면 자료 넣기 화면으로 이동하고, 카톡 대화나 상황 설명, 캡처 OCR, 녹음 전사 내용을 그대로 넣는다. `루아에게 보내기`를 누르면 별도 대기 화면으로 전환된다. 이 화면에서 루아 캐릭터와 짧은 진행 문구를 보여주며 AI 사건 접수 API를 호출한다.

AI 정리가 끝나면 싸움 일지 확인 화면으로 넘어간다. 이 화면은 레퍼런스처럼 번호가 붙은 항목 리스트를 중심으로 구성한다. 사용자는 항목을 수정, 삭제, 추가할 수 있고, 최종적으로 `이대로 판독하기`를 눌러 결과 화면으로 이동한다.

아래로 계속 콘텐츠가 길어지는 구조는 사용하지 않는다. 한 단계가 끝나면 다음 페이지로 이동한다. 뒤로 가기는 이전 단계로만 동작한다.

## 사용자 흐름

1. 홈에서 입력 방식 선택
   - `카톡 싸움 붙여넣기`
   - `증거 캡처 제출하기`
   - `현장 녹음 시작`
   - `녹음 파일 불러오기`

2. 자료 넣기
   - 화면 제목: `루아에게 보낼 자료`
   - 설명: `카톡 대화나 상황을 그대로 넣어주세요. 정리는 루아가 할게요.`
   - 주제 선택 칩은 유지한다.
   - 추가 맥락 입력은 유지하되 보조 영역으로 둔다.
   - 캡처, 녹음, 녹음 파일 플로우는 이 화면에 자동으로 텍스트를 채운다.
   - 하단 CTA: `루아에게 보내기`

3. 루아 정리 중
   - 중앙에 루아 캐릭터를 둔다.
   - 문구: `루아가 싸움 일지를 정리하고 있어요`
   - 보조 문구: `카톡 말투와 앞뒤 상황을 A/B 주장으로 나누는 중이에요.`
   - 진행 시간은 실제 API 시간에 맞추되, 최소 1초는 보여준다.
   - API 실패 시 로컬 정리로 폴백하고 다음 화면으로 이동한다.
   - 실패 안내는 막는 에러가 아니라 `간단 정리로 먼저 진행할게요.`처럼 작게 보여준다.

4. 싸움 일지 확인
   - 화면 제목: `루아가 정리한 싸움 일지`
   - 루아가 만든 항목을 번호 카드로 보여준다.
   - 예시:
     - `1. A는 답장이 늦어 서운했다고 말했어요.`
     - `2. B는 회의 중이라 답하지 못했다고 설명했어요.`
     - `3. 둘 다 사전 설명이 부족해 감정이 커졌어요.`
   - 각 항목은 수정할 수 있다.
   - 항목은 삭제할 수 있다.
   - `+ 항목 추가`로 직접 항목을 더할 수 있다.
   - `더 정확해지려면` 질문은 긴 목록이 아니라 접힌 안내로 둔다.
   - 하단 CTA: `이대로 판독하기`

5. 판독 결과
   - 기존 결과 화면을 유지하되, 상단에 `루아가 확인한 싸움 일지 기준`이라는 작은 문구를 추가한다.
   - 판독 엔진에는 원본 textarea가 아니라 사용자가 확인한 싸움 일지와 원본이 합쳐진 텍스트를 전달한다.
   - 결과 이후 보상, 공유, 판례 분석 흐름은 기존 구조를 유지한다.

## 화면별 역할

### EvidenceInputScreen

자료 넣기 전용 화면이다. 기존 `TextReview`의 주제 선택, textarea, 추가 맥락, `mediaControl`, paste handler를 이 화면으로 옮긴다. 이 컴포넌트는 사용자가 보낸 원본 자료를 관리하고, 입력이 비어 있으면 다음 단계로 보내지 않는다.

입력 화면의 책임은 자료 수집까지만이다. AI 사건 정리 결과나 판독 결과를 같은 화면 아래에 렌더링하지 않는다.

### LuaIntakeLoadingScreen

루아 정리 중 전용 화면이다. API 호출 자체는 부모 상태에서 실행하되, 화면은 루아 캐릭터, 진행 문구, 폴백 안내 상태만 보여준다.

이 화면은 사용자가 조작할 필요가 거의 없다. 뒤로 가기는 자료 넣기 화면으로 돌아간다.

### IncidentJournalScreen

싸움 일지 확인 전용 화면이다. `IncidentIntakeSummary`를 받아 `journalItems`로 변환해 보여준다. 사용자는 번호 카드 단위로 항목을 수정, 삭제, 추가한다.

이 화면에서 최종 판독 텍스트를 만든다. 원본 자료는 숨겨진 근거로 유지하고, 사용자가 확인한 일지를 앞에 둔다.

### TextReview

현재 `TextReview`는 입력, 정리, 확인, 판독 CTA를 모두 담당한다. 이번 개선에서는 `TextReview`를 새 컴포넌트들로 대체한다. 기존 `TextReview.test.tsx`는 제거하거나 새 화면 테스트로 이동한다. `App.tsx`는 더 이상 `TextReview`를 직접 렌더링하지 않고 입력 플로우 상태에 따라 `EvidenceInputScreen`, `LuaIntakeLoadingScreen`, `IncidentJournalScreen`을 렌더링한다.

## 데이터 흐름

```text
initialText + topic + extraContext + userPerspective
  -> prepareIncidentIntake()
  -> IncidentIntakeSummary
  -> journalItems
  -> buildJudgeText(summary, journalItems)
  -> analyzeWithAi()
  -> ResultScreen
```

`prepareIncidentIntake()`는 기존 AI 사건 접수 어댑터를 사용한다. 서버가 실패하면 `createLocalIncidentSummary()`를 통해 로컬 요약을 만든다.

`buildJudgeText()`는 새 유틸로 분리한다. 입력은 `IncidentIntakeSummary`, 수정된 `journalItems`, 원본 텍스트, 추가 맥락이다. 출력은 판독 엔진에 전달할 문자열이다.

## 상태 전이

입력 플로우 내부 상태는 다음처럼 둔다.

- `review`: 자료 넣기
- `intake-loading`: 루아 정리 중
- `journal`: 싸움 일지 확인
- `result`: 기존 판독 결과

뒤로 가기 동작은 다음과 같다.

- `intake-loading`에서 뒤로 가기: `review`
- `journal`에서 뒤로 가기: `review`
- `result`에서 다시 판독하기: 홈
- 앱 브라우저 뒤로 가기: 현재 단계 기준 이전 화면, 홈에서는 앱 내부 홈 유지

## 에러 처리

입력 없음은 자료 넣기 화면에서 막는다. 메시지는 `루아에게 보낼 내용을 먼저 넣어주세요.`로 한다.

AI 사건 접수 실패는 사용자 흐름을 막지 않는다. 로컬 정리로 폴백하고 싸움 일지 확인 화면으로 이동한다. 이때 상단 또는 작은 안내로 `AI 정리가 불안정해서 간단 정리로 먼저 진행할게요.`를 보여준다.

판독 실패는 싸움 일지 확인 화면에 남긴다. 사용자는 항목을 고치거나 다시 판독을 시도할 수 있다.

OCR, 녹음, 녹음 파일 변환 실패는 기존 방식처럼 자료 넣기 화면에서 직접 입력을 안내한다.

## UX 원칙

- 한 화면에는 하나의 주요 행동만 둔다.
- 강한 CTA는 하단에 하나만 둔다.
- 다음 단계 결과를 같은 화면 아래에 이어 붙이지 않는다.
- 사용자가 확인한 싸움 일지를 판독 기준으로 삼는다.
- 루아 캐릭터는 대기 화면에서 가장 강하게 사용한다.
- 문구는 해요체를 유지하고, 실제 법적 판단처럼 보이지 않게 한다.
- 무료 판독은 재미용 판독이라는 안내는 유지한다.

## 테스트 기준

- 카톡 붙여넣기 선택 시 `자료 넣기 -> 루아 정리 중 -> 싸움 일지 확인 -> 판독 결과` 순서로 이동한다.
- 캡처/OCR 결과가 자료 넣기 화면에 채워지고 같은 플로우로 합류한다.
- 현장 녹음 후 직접 입력도 같은 플로우로 합류한다.
- 녹음 파일 전사 결과가 자료 넣기 화면에 채워지고 같은 플로우로 합류한다.
- AI 사건 접수 API가 실패해도 로컬 정리된 싸움 일지 확인 화면으로 이동한다.
- 싸움 일지 확인 화면에서 항목 수정, 삭제, 추가가 가능하다.
- `이대로 판독하기`를 누르면 사용자가 확인한 일지 기준 텍스트가 `analyzeWithAi()`로 전달된다.
- 뒤로 가기는 현재 단계 기준 이전 페이지로 이동한다.
- 모바일 viewport에서 하단 CTA와 입력 영역이 겹치지 않는다.
- 각 화면의 텍스트가 버튼이나 카드 밖으로 넘치지 않는다.

## 구현 범위

이번 작업에서 포함한다.

- 입력 플로우를 페이지 전환형 상태로 분리
- `EvidenceInputScreen`, `LuaIntakeLoadingScreen`, `IncidentJournalScreen` 추가
- `buildJudgeText()` 유틸 추가
- 기존 텍스트, 캡처, 녹음, 녹음 파일 입력 경로를 새 플로우로 연결
- AI 사건 접수 폴백 UX 정리
- 관련 단위 테스트와 App 플로우 테스트 수정

이번 작업에서 제외한다.

- 새로운 AI 모델 선택 정책 변경
- 핫 Battle 상세 UX 재설계
- 보상 추천 화면 재설계
- 판례 분석 결제 UX 재설계
- Supabase 스키마 변경

## 완료 기준

사용자가 홈에서 입력 방식을 하나 고른 뒤, 긴 스크롤 없이 한 페이지씩 이동하며 판독 결과까지 도달할 수 있어야 한다. 루아가 정리한 싸움 일지를 사용자가 확인했다는 감각이 있어야 하고, 판독 결과는 그 확인된 일지를 기준으로 생성되어야 한다.
