# 누가 잘못 AI 레드 리포트 모션 UI 설계

## 배경

현재 `누가 잘못 AI`는 기능 흐름이 완성된 Toss WebView 미니앱 MVP다. 사용자는 텍스트, 캡처, 녹음 흐름으로 싸움 내용을 넣고 무료 판독 결과를 본 뒤, 보상 추천과 990원 판례 판독 준비 화면을 확인할 수 있다.

이번 UI/UX 개편은 기능 흐름을 유지하면서 첫인상과 결과 표현을 강화한다. 사용자는 시각 시안에서 **C. 분석 리포트 모션**을 선택했고, 메인 화면은 첨부 참고 이미지의 강한 빨강, 큰 숫자, 폰 프레임, 콘텐츠가 꽉 찬 프로모션 무드를 선호했다.

참고 시안:

- 브레인스토밍 서버: `http://localhost:52739`
- 시안 파일: `.superpowers/brainstorm/44057-1779266463/content/main-red-report-motion.html`

## 목표

- 현재 앱의 입력/분석/결과 기능은 유지한다.
- 첫 화면을 강한 레드 비주얼과 큰 그래픽 요소가 있는 “사건 접수” 화면으로 바꾼다.
- 모션은 **분석 리포트형**을 기본으로 한다.
- 결과 화면에서 퍼센트 바, 근거 카드, 보상 추천 카드가 순차적으로 등장한다.
- 루아 AI 캐릭터는 메인 화면의 중심 비주얼로 더 크게 사용한다.
- 망치 모션은 결과 확정 순간에 짧게 1회만 사용한다.
- Toss 미니앱답게 가독성, 클릭 안정성, 과하지 않은 움직임을 유지한다.

## 비목표

- 기능 흐름을 바꾸지 않는다.
- 실제 결제, 판례 서버, Toss Shopping 딥링크를 새로 연결하지 않는다.
- 메인 화면을 랜딩 페이지처럼 길게 만들지 않는다.
- 참고 이미지의 게임 캐릭터 IP나 레이아웃을 직접 복제하지 않는다.
- 반복적이고 시끄러운 애니메이션을 넣지 않는다.

## 시각 방향

전체 톤은 **레드 리포트 재판 UX**다.

- 배경: 강한 레드 그라데이션을 홈 화면 중심으로 사용한다.
- 카드: 실제 기능 조작 영역은 Toss식 흰색 카드와 8px 안팎의 절제된 radius를 유지한다.
- 그래픽: `AI`, `62`, `990` 같은 큰 숫자/텍스트를 배경 그래픽처럼 사용한다.
- 캐릭터: 루아 AI를 홈의 첫 번째 시각 신호로 크게 배치한다.
- 정보 밀도: 프로모션 이미지처럼 강한 첫인상을 주되, 입력 버튼은 명확히 읽히게 한다.

## 화면별 설계

### 홈 화면

현재 홈의 4개 입력 방식은 유지한다.

- 카톡 싸움 붙여넣기
- 증거 캡처 제출하기
- 현장 녹음 시작
- 녹음 파일 불러오기

홈 화면 구조:

1. 상단 브랜드: `누가 잘못 AI`
2. 서브카피: `루아 AI가 판독해드립니다`
3. 히어로 카피: `사건을 접수해 주세요`
4. 보조 카피: `대화, 캡처, 녹음을 증거로 제출하면 루아 AI가 판독해드려요.`
5. 중앙 캐릭터: 루아 AI 큰 이미지
6. 배경 그래픽: `AI`, `62`, `990` 중 1~2개를 화면 안쪽에 크게 배치
7. 입력 카드: `증거 접수` 느낌으로 순차 등장

홈 모션:

- 화면 진입 시 캐릭터가 160ms 정도 늦게 부드럽게 떠오른다.
- 입력 카드 4개는 60ms 간격으로 아래에서 위로 등장한다.
- 배경 숫자는 아주 느린 opacity/translate 움직임만 가진다.
- 사용자가 `prefers-reduced-motion: reduce`인 경우 모든 진입 모션을 제거한다.

### 텍스트 확인 화면

현재 구조를 유지하되, “증거 기록지” 느낌을 더한다.

- 제목: `루아 AI가 판독할 대화`
- 보조 문구: `판독 전에 증거 내용을 확인하고 고칠 수 있어요.`
- textarea는 흰 카드 안에 유지한다.
- 미묘한 상단 라벨이나 작은 사건번호 느낌의 텍스트를 넣을 수 있다.

이 화면에는 과한 모션을 넣지 않는다. 텍스트 편집 안정성이 우선이다.

### 결과 화면

결과 화면은 C안의 핵심이다. “분석 리포트”처럼 순서대로 읽히게 한다.

구조:

1. 오늘의 판결 카드
2. A/B 퍼센트 바
3. 근거 3개 카드
4. 한 줄 조언
5. 승소 보상 추천
6. 990원 판례 판독 미리보기

결과 모션:

- 결과 화면 진입 시 `오늘의 판결` 카드가 먼저 등장한다.
- A/B 퍼센트 바는 0에서 결과 비율까지 차오른다.
- 근거 3개는 80ms 간격으로 순차 등장한다.
- 보상 추천과 판례 미리보기는 아래쪽에서 살짝 등장한다.
- 루아 AI 망치 모션은 판결 카드 등장 직후 1회만 실행한다.
- 모든 모션은 180~420ms 범위에서 끝낸다.

## 컴포넌트 설계

새로 만들거나 분리할 수 있는 UI 단위:

- `MotionHomeHero`: 레드 히어로, 캐릭터, 큰 배경 숫자, 홈 카피
- `EvidenceMethodCard`: 입력 방식 카드
- `AnimatedPercentBar`: A/B 퍼센트 결과 바
- `VerdictSummaryCard`: 오늘의 판결 카드
- `ResultReasonCard`: 근거 카드
- `MotionGate`: `prefers-reduced-motion`을 반영하는 작은 hook 또는 CSS utility

기존 기능 모듈은 유지한다.

- 분석 로직은 `ruleBasedAnalyzer`를 그대로 사용한다.
- 보상 추천은 `rewardAdapter`를 그대로 사용한다.
- 프리미엄/판례 placeholder는 `premiumAdapter`, `precedentAdapter`를 그대로 사용한다.

## 스타일 원칙

- 레드는 홈과 핵심 강조 영역에만 강하게 사용한다.
- 긴 텍스트와 입력 폼은 흰 배경 카드로 유지한다.
- 버튼은 44px 이상의 터치 영역을 유지한다.
- 텍스트는 viewport width 기반으로 줄이지 않는다.
- 카드 안 텍스트가 넘치지 않게 `overflow-wrap`과 안정적인 grid/flex 크기를 유지한다.
- 모션은 `transform`, `opacity` 위주로 구현한다.
- layout shift를 일으키는 height/width 애니메이션은 쓰지 않는다.

## 안전 및 접근성

- `prefers-reduced-motion: reduce`에서는 진입/반복 애니메이션을 제거한다.
- 중요한 결과 텍스트는 애니메이션 없이도 DOM에 존재하고 읽힌다.
- 입력 카드와 버튼의 accessible name은 현재처럼 명확하게 유지한다.
- 보상 추천과 프리미엄 상태는 `role="status"`를 유지한다.
- 안전 결과에서는 보상 추천과 판례 미리보기를 계속 숨긴다.

## 테스트 기준

- 홈 화면에 `누가 잘못 AI`, `루아 AI가 판독해드립니다`, `사건을 접수해 주세요`가 보인다.
- 홈 입력 카드 4개가 모두 보이고 클릭 가능한 상태다.
- 캐릭터 이미지가 렌더링된다.
- 결과 화면에 퍼센트 바가 A/B 비율을 반영한다.
- 근거 3개가 모두 보인다.
- 보상 추천과 판례 미리보기는 normal 결과에서만 보인다.
- safety 결과에서는 보상 추천과 판례 미리보기가 보이지 않는다.
- `prefers-reduced-motion` 환경에서 주요 테스트가 깨지지 않는다.
- `npm test`, `npm run lint`, `npx tsc -b --pretty false`, `npm run build`가 통과한다.

## 승인된 결정

- 모션 방향은 **C. 분석 리포트 모션**이다.
- 메인 화면은 첨부 이미지의 **강한 빨강, 큰 숫자, 폰 안 콘텐츠가 꽉 찬 느낌**을 참고한다.
- 실제 앱에서는 그 무드를 Toss식 카드 UI와 섞어 가독성을 지킨다.
- 기능 흐름은 유지하고 UI/UX와 모션만 바꾼다.
