# 루아 AI 디테일 UX 고도화 설계

## 목적

현재 홈 화면은 유지한다. UX 고도화의 대상은 홈 이후의 디테일 페이지와 첫 방문 인트로다. 목표는 기능을 더 늘리는 것이 아니라, 사용자가 `루아 법정 쇼 -> 홈 -> 판정 결과 -> 보상/댓글/판례/판정방` 흐름을 자연스럽게 따라가도록 만드는 것이다.

## 범위

포함한다.

- 첫 방문 전용 루아 법정 쇼 인트로
- 판정 결과 상세
- 루아 보상 상담소
- 핫 Battle 상세/댓글
- 60초 폭파 판정방
- 이의제기/판례 분석

포함하지 않는다.

- 홈 화면 구조 변경
- 새로운 서버 기능 추가
- Toss 쇼핑, Toss IAP, Supabase의 실제 운영 credential 설정
- 법률 자문처럼 보이는 문구나 결과 보장 표현

## 핵심 UX 원칙

1. 홈은 기능 허브로 유지한다.
2. 첫 방문 인트로가 앱의 세계관과 캐릭터성을 담당한다.
3. 각 디테일 페이지는 한 화면에서 하나의 목적만 강하게 보여준다.
4. 보상, 댓글, 판례 분석은 결과 화면을 가리지 않고 다음 행동으로 분리한다.
5. 공개/저장/서버 전송이 있는 기능은 화면에서 명확히 설명한다.

## 첫 방문 인트로

인트로는 첫 방문에만 보여준다. 사용자가 `판정 시작하기`를 누르면 첫 방문 완료 상태를 로컬에 저장하고 홈으로 이동한다. 저장이 실패해도 홈 이동은 가능해야 한다.

인트로의 역할은 기능 선택이 아니라 브랜드 경험이다. 큰 루아 캐릭터, 법정 배경, 짧은 조명/망치/캐릭터 모션, 한 문장 안내를 보여준다. 홈의 입력 카드와 기능 목록은 반복하지 않는다.

추천 문구:

> 증거를 제출하세요. 누가 선 넘었는지 루아가 판독해드릴게요.

## 판정 결과 상세

판정 결과 상세는 모든 후속 행동의 기준점이다. 상단에는 `A가 72% 선넘었어요` 같은 결과 문구를 가장 크게 보여준다. 망치 판정 이펙트와 A/B 비율을 함께 배치해 결과를 즉시 이해하게 한다.

결과 아래에는 근거 3개와 루아의 한 줄 조언만 둔다. 사용자가 결과를 납득한 뒤 다음 행동을 고르게 한다.

주요 액션은 두 개만 크게 둔다.

- 왼쪽 금색 버튼: `보상받기`
- 오른쪽 기본 버튼: `다시 판독하기`

댓글/공유는 탭이 아니라 게시판형 댓글 영역으로 결과 아래에 배치한다. 입력창, 최신 댓글, 좋아요, 공유를 같은 커뮤니티 섹션에 묶는다.

이의제기/판례 분석은 기본 펼침 상태로 두지 않는다. 하단에 빨간 CTA로 배치하고, 클릭하면 결제 전 확인 화면으로 넘어간다.

추천 CTA 문구:

> 억울하면 유사 판례로 한 번 더 따져보기

## 루아 보상 상담소

보상 상담소는 이긴 사람이 받고 싶은 것을 말하면 루아가 잘못 정도에 맞춰 토스 쇼핑 후보를 골라주는 화면이다. 진입 직후 루아 캐릭터와 말풍선을 보여주고, 판정 결과를 이어받아 설명한다.

예시:

> A가 72% 선넘었으니, 너무 과하지 않게 보상 후보를 골라볼게요.

사용자는 받고 싶은 보상을 한 줄로 입력한다. 예시는 `달달한 거`, `커피`, `귀여운 거`, `작은 선물`처럼 가볍고 직관적인 표현을 쓴다.

추천 결과는 토스 쇼핑형 상품 카드 3개로 보여준다. 각 카드는 상품 이미지 영역, 상품명, 가격대, 추천 이유 한 줄을 가진다.

가격대 원칙:

- 50~60%: 5천원대
- 60~80%: 1~2만원대
- 80% 이상: 3만원 이상

후보 톤:

- 가벼운 사과
- 적정 보상
- 확실한 사과

사용자는 후보를 체크해서 상대에게 보낼 수 있다. 실제 Toss 쇼핑 API 연결 전에는 검색 URL 또는 추천 URL 어댑터를 사용한다. 화면에는 구매 강제가 아니라 보상 후보 추천임을 유지한다.

## 핫 Battle 상세/댓글

핫 Battle 상세는 구경하는 사용자가 싸움 맥락을 읽고 댓글을 남기는 게시판형 상세다.

표시 순서:

1. 순위, 좋아요, 댓글 수, 공유
2. 싸움 제목
3. 대화 내용
4. 루아 판정 결과와 근거 요약
5. 댓글 쓰기
6. 댓글 리스트

대화 내용을 먼저 보여준 뒤 판정을 보여준다. 사용자가 맥락 없이 결과만 보는 느낌을 줄이기 위해서다.

댓글 UI는 게시판형으로 만든다. 댓글 입력창, 최신순/공감순 정렬, 댓글 리스트를 둔다. 좋아요/싫어요보다 `선넘었어요` 공감 표현을 우선 검토한다.

공개된 Battle의 대화는 원문 전체가 아니라 익명화 또는 요약된 내용이어야 한다. 이름, 전화번호, 주소 등 개인정보는 표시하지 않는다.

## 60초 폭파 판정방

판정방은 채팅은 자연스럽고, 결과 전환은 극적으로 만든다.

입장 전에는 초대장처럼 보이게 하고 닉네임을 입력받는다. 첫 두 명은 A/B 당사자, 세 번째부터는 관전자다.

카운트다운은 A/B 두 명이 모두 입장해야 시작한다. 혼자 있거나 관전자만 들어온 상태에서는 대기 상태를 유지한다.

채팅 중에는 Telegram 같은 말풍선 중심 레이아웃을 사용한다. 관전자는 읽기만 가능하고, 입력창 대신 `관전 중이에요` 상태를 보여준다.

타이머가 5초 이하가 되면 법정 조명, 망치, 카운트다운 등 긴장감을 높이는 모션을 넣는다.

폭파 후에는 대화가 그냥 사라지는 느낌을 피한다. 루아가 `증거는 정리했어요` 같은 톤으로 원문 삭제를 설명하고 결과 화면으로 전환한다. 결과는 자동 공개하지 않고, 사용자가 공유를 선택해야 핫 Battle이나 댓글 흐름으로 이어진다.

## 이의제기/판례 분석

이의제기/판례 분석은 결제형 기능이므로 재미보다 신뢰와 오해 방지가 우선이다.

결과 화면 하단에는 빨간 CTA를 접힌 상태로 둔다.

추천 문구:

> 억울하면 유사 판례로 한 번 더 따져보기

클릭하면 결제 전 확인 화면으로 이동한다. 결제 전 확인 화면에는 아래 내용을 반드시 보여준다.

- 가격: 990원
- 서버로 전송되는 정보: 판독 대화 텍스트
- 결과 성격: 유사 판례 참고이며 법률 상담이 아님
- 유사 판례가 없을 수도 있음

사용자가 전송과 참고용 성격에 동의하기 전에는 결제/분석 버튼을 활성화하지 않는다.

분석 결과 화면은 유사 판례 2~3개, 왜 참고되는지 한 줄 근거, 기존 판정과 달라진 점을 보여준다. 결제가 취소된 경우, 결제는 성공했지만 분석이 실패한 경우, 유사 판례가 없는 경우의 메시지를 분리한다.

피해야 할 문구:

- 제대로 따져드립니다
- 법적으로 누가 맞는지 판단해드립니다
- 결제하면 결과가 뒤집힙니다

안전한 문구:

- 유사 판례를 참고해 한 번 더 분석해드려요
- 실제 법률 상담이 아닌 참고용 분석이에요
- 결과가 달라질 수도, 유지될 수도 있어요

## 데이터와 상태

첫 방문 인트로는 로컬 저장소에 완료 여부를 저장한다. 저장 실패 시에도 홈 이동을 막지 않는다.

결과 화면은 기존 `JudgmentResult`를 기준으로 보상, 댓글, 판례 분석으로 분기한다. 결과 원문은 사용자가 공유하거나 판례 분석에 동의하기 전에는 공개 저장하지 않는다.

보상 상담소는 판정 결과의 책임 비율과 사용자의 보상 키워드를 입력으로 받아 추천 후보를 만든다.

핫 Battle 상세는 공유된 결과, 익명화된 대화 요약, 댓글, 좋아요를 사용한다.

판정방은 방 ID, 방 접근 키, 참여자 역할, 메시지, 만료 시간을 사용한다. 폭파 후 원문 메시지는 삭제되고 결과만 남긴다.

## 오류와 빈 상태

- 인트로 저장 실패: 홈으로 이동하고 다음 방문에 인트로가 다시 나올 수 있음을 허용한다.
- 보상 추천 실패: 검색 키워드를 바꾸거나 다시 시도하도록 안내한다.
- 쇼핑 URL 미연결: 검색 후보임을 표시하고 앱이 멈추지 않게 한다.
- 댓글 저장 실패: 댓글 입력값을 유지하고 다시 시도하게 한다.
- 판정방 링크 오류: 만료되었거나 잘못된 초대 링크임을 보여주고 홈으로 돌아가게 한다.
- 판례 결제 취소: 결제 취소 상태로 돌아오고 금액이 청구되지 않았음을 안내한다.
- 판례 분석 실패: 결제 실패와 분석 실패 메시지를 분리한다.

## 테스트 기준

단위/컴포넌트 테스트:

- 첫 방문 인트로가 완료 전에는 보이고 완료 후에는 홈으로 바로 가는지 확인한다.
- 판정 결과 화면에 보상받기/다시 판독하기가 큰 주요 액션으로 보이는지 확인한다.
- 보상 상담소가 책임 비율에 따라 가격대를 다르게 추천하는지 확인한다.
- 핫 Battle 상세가 대화 내용, 판정, 댓글 순서로 렌더링되는지 확인한다.
- 판정방은 A/B 모두 입장 전에는 카운트가 시작되지 않는지 확인한다.
- 판례 분석은 동의 전 결제/분석 버튼이 비활성인지 확인한다.

수동/브라우저 검수:

- 모바일 폭에서 결과 화면 하단 버튼과 댓글 영역이 겹치지 않는지 확인한다.
- 보상 카드의 텍스트가 상품 카드 안에서 깨지지 않는지 확인한다.
- 판정방에서 키보드가 열릴 때 입력창이 가려지지 않는지 확인한다.
- Toss WebView에서 뒤로가기 동작이 홈/상세 흐름과 충돌하지 않는지 확인한다.

## 구현 순서

1. 첫 방문 인트로
2. 판정 결과 상세
3. 루아 보상 상담소
4. 핫 Battle 상세/댓글
5. 60초 폭파 판정방
6. 이의제기/판례 분석

이 순서는 사용자가 승인한 UX 정리 순서를 따른다.
