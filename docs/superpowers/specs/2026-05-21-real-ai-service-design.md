# 누가 잘못 AI 실제 AI 서비스 전환 설계

## 목적

`누가 잘못 AI`를 로컬 규칙 기반 MVP에서 실제 AI, 서버 저장, 결제, 판례 분석, 공개 게시판을 갖춘 Toss 미니앱 서비스로 전환한다. 앱의 핵심 재미는 `누가 몇 퍼센트 선넘었는지`를 빠르게 보여주는 데 있고, 실제 서비스에서는 비용, 개인정보, 결제 검증, 공개 댓글 리스크를 함께 제어해야 한다.

이 설계는 구현 순서를 정하기 전의 제품/기술 기준이다. 홈 UI의 세부 배치는 별도 UX 작업으로 다루고, 여기서는 실제 운영 가능한 백엔드와 데이터 흐름을 정의한다.

## 확정된 제품 방향

- 무료 판독도 실제 서버 AI를 사용한다.
- 무료 판독은 빠른 저비용 모델을 사용한다.
- 990원 판례 판독은 더 강한 모델과 판례 검색 결과를 함께 사용하는 RAG 구조로 만든다.
- 아키텍처는 Vercel API와 Supabase를 함께 쓰는 혼합형으로 간다.
- 기본 저장 정책은 사용자 선택 저장형이다.
- 무료 사용량은 하루 3회이며, 공유 결과 생성에 성공하면 하루 추가 1회를 준다.
- 핫 Battle은 원문을 공개하지 않고, AI가 익명화한 요약과 판정 결과만 공개한다.
- 실시간 폭발 판정방은 A/B가 모두 들어오면 60초 카운트를 시작하고, 즉시 판정 또는 자동 판정으로 종료한다.
- AI 톤은 재미있는 판정과 부드러운 완충을 함께 쓴다.

## 전체 구조

```text
Toss Mini App
  -> Vercel API
      -> 무료 AI 판독
      -> 유료 판례 AI 판독
      -> Toss IAP 결제 검증
      -> 사용량 제한
      -> 개인정보/안전 필터
  -> Supabase
      -> 실시간 폭발 판정방
      -> 판정 결과 저장
      -> 핫 Battle
      -> 댓글/좋아요/신고
      -> 익명 사용자 사용량 기록
  -> OpenAI
      -> 무료 빠른 모델
      -> 유료 고급 모델
  -> Precedent Index
      -> 판례 검색
      -> 유료 판례 RAG 근거
```

프론트는 OpenAI 키, 결제 검증 키, 판례 entitlement secret을 절대 갖지 않는다. 모든 AI 호출과 결제 검증은 Vercel API에서 처리한다. Supabase anon key는 프론트에 노출될 수 있으므로 직접 테이블 접근을 막고 RPC 중심으로 접근한다.

## 사용자 흐름

### 무료 AI 판독

1. 사용자가 텍스트, 캡처, 녹음 흐름으로 대화를 입력한다.
2. `나는 누구예요?` 단계에서 첫 번째 사람, 두 번째 사람, 잘 모르겠어요 중 하나를 고른다.
3. 프론트가 입력 텍스트와 사용자 관점을 Vercel API로 보낸다.
4. 서버가 길이 제한, 무료 횟수, 민감정보, 안전 위험을 확인한다.
5. 서버가 빠른 AI 모델로 JSON 판정 결과를 생성한다.
6. 결과 화면은 `상대방이 72% 선넘었어요`처럼 사용자 관점으로 보여준다.
7. 공유용/핫 Battle용 데이터는 `A/B` 익명 기준으로 변환한다.

### 990원 판례 판독

1. 결과 화면에서 사용자가 `억울하면 판례로 다시 따지기`를 누른다.
2. 결제 전 서버 전송과 참고용 분석 고지를 보여준다.
3. Toss IAP 결제를 실행한다.
4. 서버가 Toss 결제 성공 여부를 검증한다.
5. 서버가 entitlement를 발급한 뒤 판례 검색과 고급 AI 분석을 실행한다.
6. 결과는 유사 판례, 쟁점, 책임 판단, 반박 포인트, 화해 제안을 포함한다.
7. 법률 상담이 아니라 참고용 분석임을 화면에 명확히 표시한다.

### 실시간 폭발 판정방

1. 사용자가 방을 만들고 닉네임을 입력한다.
2. 초대 링크를 공유한다.
3. 첫 두 명은 A/B 당사자로 입장하고, 세 번째부터는 관전자다.
4. A/B가 모두 입장하면 60초 카운트가 시작된다.
5. A/B 중 한 명이 `지금 판정`을 누르면 즉시 종료할 수 있다.
6. 60초가 지나면 자동으로 종료된다.
7. 서버 AI가 대화를 판독한다.
8. 원문 메시지는 삭제하고 결과만 남긴다.
9. 결과 공개 이후 댓글과 좋아요 참여가 가능하다.

### 핫 Battle

1. 사용자가 결과 화면에서 `사람들한테 물어보기`를 선택한다.
2. 서버가 공개용 제목, 쟁점 요약, 익명화된 대화 요약을 만든다.
3. 공개 저장 전 민감정보 필터를 통과해야 한다.
4. 핫 Battle에는 결과, 요약, 근거, 조언, 댓글 수, 좋아요 수를 보여준다.
5. 댓글은 금칙어, 연락처, 주소, 실명성 표현을 필터링한다.

## AI 결과 스키마

서버 AI는 프론트가 바로 렌더링할 수 있는 JSON만 반환해야 한다.

```ts
type AiJudgmentResult = {
  winner: "A" | "B" | "draw";
  blamedParty: "A" | "B" | "both" | "unknown";
  partyAPercent: number;
  partyBPercent: number;
  verdict: string;
  userPerspectiveVerdict: string;
  reasons: [string, string, string];
  advice: string;
  tone: "light" | "serious" | "safety";
  safetyLevel: "normal" | "caution" | "urgent";
  rewardTier: "small" | "medium" | "large";
  publicTitle: string;
  issueSummary: string;
  anonymizedDialogueSummary: [string, string];
  shareSummary: string;
};
```

`partyAPercent + partyBPercent`는 100이 되도록 서버에서 보정한다. 위험 상황이 감지되면 판정 승패보다 안전 안내를 우선한다.

## 저장 정책

무료 일반 판독은 원문을 영구 저장하지 않는다. AI 분석 요청 중에만 사용하고, 결과도 기본 저장하지 않는다. 사용자가 최근 판정 저장, 공유, 핫 Battle 공개를 선택한 경우에만 결과를 저장한다.

실시간 폭발 판정방의 원문 메시지는 판정 직후 삭제한다. 저장되는 것은 판정 결과, 공개용 익명 요약, 댓글/좋아요 집계뿐이다.

유료 판례 분석은 결제 검증과 고객 문의 대응을 위해 결제 식별자, 분석 결과, 사용한 판례 식별자는 저장한다. 원문 대화는 기본적으로 저장하지 않고, 사용자가 최근 판정 저장에 동의한 경우에만 제한 기간 저장한다.

## Supabase 데이터

필요한 핵심 테이블은 다음과 같다.

- `anonymous_users`: 익명 사용자 키, 생성일, 제한 상태
- `daily_ai_usage`: 날짜별 무료 사용량, 공유 보너스 사용량
- `judgment_results`: 저장 선택된 판정 결과
- `public_battles`: 핫 Battle 공개 데이터
- `result_comments`: 댓글
- `result_likes`: 좋아요
- `result_reports`: 신고
- `judgment_rooms`: 실시간 판정방
- `room_participants`: A/B/관전자 입장 상태
- `room_messages`: 폭발 전 임시 메시지
- `premium_orders`: Toss 결제 검증 결과
- `premium_precedent_reports`: 유료 판례 분석 결과

프론트에서 직접 테이블을 읽고 쓰지 않는다. 공개 기능은 RPC 또는 서버 API를 통해서만 접근한다.

## Vercel API

필요한 API는 다음과 같다.

- `POST /api/ai/free-judgment`
  - 무료 AI 판독
  - 입력 길이 제한, 일일 사용량 제한, 안전 필터 적용

- `POST /api/ai/public-summary`
  - 핫 Battle 공개용 익명 요약 생성
  - 공개 저장 전 민감정보 필터 적용

- `POST /api/payments/precedent-entitlement`
  - Toss IAP 결제 서버 검증
  - 성공 시 짧은 TTL의 entitlement 발급

- `POST /api/ai/precedent-judgment`
  - entitlement 확인
  - 판례 검색
  - 고급 AI 판례 분석

- `POST /api/moderation/comment`
  - 댓글 공개 전 필터

## 결제 정책

SKU 기본값은 `precedent-verdict-990`로 둔다. 클라이언트 이벤트만 믿지 않고, Vercel API에서 Toss 결제 서버 검증을 수행한다. 서버 검증이 성공한 주문에만 entitlement를 발급한다. entitlement는 짧은 TTL을 가지며, 판례 분석 API에서만 사용할 수 있다.

결제 실패, 검증 실패, 판례 없음, AI 실패는 서로 다른 사용자 메시지로 분리한다. 결제만 성공하고 분석이 실패한 경우에는 재시도 또는 환불 문의로 이어질 수 있게 주문 식별자를 남긴다.

## 보안과 심사 리스크

- OpenAI API key와 결제 secret은 서버 env에만 둔다.
- Supabase anon key로 직접 테이블 접근이 불가능해야 한다.
- 모든 공개 결과는 원문이 아니라 익명 요약만 사용한다.
- 댓글은 연락처, 주소, 실명, 욕설, 위협 표현을 필터링한다.
- 긴 대화, 반복 요청, 같은 기기 과다 요청은 제한한다.
- 안전 위험 대화는 승패 판정 대신 도움 안내로 전환한다.
- 법률 상담처럼 보이지 않게 판례 분석 화면에 참고용 고지를 고정 노출한다.

## 구현 우선순위

1. 무료 AI 판독 API와 프론트 adapter를 만든다.
2. 사용자 관점 선택 단계를 입력 확인 화면에 추가한다.
3. 일일 3회 무료와 공유 보너스 1회 제한을 Supabase에 저장한다.
4. 핫 Battle 공개 저장을 서버 요약 기반으로 바꾼다.
5. 댓글/좋아요/신고를 서버 저장으로 완성한다.
6. Toss IAP 서버 검증과 entitlement를 완성한다.
7. 판례 RAG 분석 API를 유료 플로우에 연결한다.
8. 실시간 방 종료 시 로컬 rule-based 대신 서버 AI 판독을 호출한다.
9. OCR/Tesseract 코드를 지연 로딩해 첫 화면 번들을 줄인다.
10. Toss 실기기에서 결제, 공유, 뒤로 가기, 링크 진입, 안전 영역을 검수한다.

## 테스트 기준

- 무료 판독은 사용량이 남아 있을 때 결과를 반환한다.
- 무료 판독 3회 이후에는 제한 메시지를 반환한다.
- 공유 결과 생성 성공 시 하루 1회 보너스가 지급된다.
- 핫 Battle 공개 데이터에는 원문 대화가 포함되지 않는다.
- 댓글 필터는 연락처와 주소성 텍스트를 차단한다.
- Toss 결제 검증 실패 시 판례 분석 API가 실행되지 않는다.
- entitlement가 없거나 만료되면 판례 분석 API는 거절한다.
- 실시간 방은 A/B 입장 후 카운트가 시작되고 판정 후 메시지가 삭제된다.
- 안전 위험 대화는 일반 승패 판정 대신 안전 안내를 반환한다.
- Vercel build, unit test, Supabase RPC 테스트가 통과한다.
